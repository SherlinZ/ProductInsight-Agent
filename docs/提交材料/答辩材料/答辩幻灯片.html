Rework 路由防循环 · v0.3 补丁

Rework 听起来高大上，但工程上最容易掉链子。3 大坑：

3 大坑

1. 无限循环：旧 rework 不会"老化"，新 rework 不断叠加

2. 状态污染：pending 和 running 状态混淆

3. 资源耗尽：没有 retry 上限，OOM 风险

修复 (v0.3 Step 1 稳定性补丁)

def route_after_review(): pending_rework = [r for r in rework_queue if r.status == "pending" and r.retry_count < MAX_ITERATIONS] for r in pending_rework: r.status = "running" r.retry_count += 1 return pending_rework

"这不是临场 hack，是设计上的防御性编程。"

超长上下文分片 · P0-5

问题：当 evidence 超过 LLM context window 会怎样？

实测场景

10 个产品 × 30 evidence = 300 条 evidence，每条 ~300 tokens，总 9 万 tokens，远超 8K context

错误方案：截断（丢失关键信息）→ 必然幻觉

正确方案 (P0-5)

def _chunk_evidence_for_llm(evidence_items, max_tokens=8000): """按 product 分组，保持相关 evidence 在同一批。""" by_product = defaultdict(list) for ev in evidence_items: by_product[ev.product_id].append(ev) chunks = [] current_chunk, current_tokens = [], 0 for product_id, evs in by_product.items(): pt = sum(estimate_tokens(e) for e in evs) if current_tokens + pt > max_tokens: chunks.append(current_chunk) current_chunk, current_tokens = [], 0 current_chunk.extend(evs) current_tokens += pt return chunks

核心设计：按 product 分组（不是随机切分）| 保持 schema 维度连贯 | 无截断
