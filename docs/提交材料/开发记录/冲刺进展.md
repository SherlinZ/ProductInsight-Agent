## 一、项目是什么

**ProductInsight Agent**（产品：竞品分析 Agent 工作台）是一个演示型应用，展示一个完整、可回放的**多 Agent 协作竞品分析工作流**。

核心原则：

```
No evidence, no claim.    ← 没有证据，不能生成断言（Claim）。
No schema, no report.      ← 没有 Schema 结构，不能出报告。
No review, no no publish.  ← 没有质检通过，不能发布。
```

**典型使用场景**：分析多个 AI Agent 平台（如 Dify、Coze、FastGPT、Flowise），对比它们的工作流编排能力、定价模式、目标用户画像，生成带证据溯源的竞品分析报告。

**技术定位**：这是一个**演示系统**，而非生产级产品。LLM 调用真实（用火山引擎 Doubao API），数据采集逻辑部分真实（可真实抓取网页），但整体以功能演示为主。

## 五、核心概念详解

### 5.1 数据模型（Evidence → Claim → Report 链路）

```
URL 来源（sources）
    ↓ Collector Agent 采集
页面快照（snapshots）
    ↓ Evidence Extractor 抽取
证据片段（evidence_items）
    ↓ Evidence Evaluator 评分
可信证据（usable_for_claim = true）
    ↓ Analyst Agent 分析
结构化事实（facts）
    ↓ Analyst Agent 生成
断言草稿（claim_drafts）
    ↓ Reviewer Agent 质检
已签声明（signed_claims） + 待返工（rework_required_claims）
    ↓ Writer Agent 生成
竞品分析报告（report）
```

**关键原则**：
- Evidence 必须通过 `evidence_evaluator.py` 的 6 维度评分（relevance、authority、freshness、schema_fit、information_density、no_noise）且 `usable_for_claim = True` 才能被 Claim 引用。
- Claim 必须被 Reviewer Agent 签署（signed）才能写入报告。
- 报告写作时，Evidence Contract（`evidence_gap.py`）会对每个分析维度设定来源白名单（如 `pricing_model` 维度禁止 `web_page` 来源），防止劣质来源支撑关键断言。

### 5.2 Schema（分析维度）

系统使用 **3 个核心 Schema** 组织分析维度：

| Schema | 中文名 | 包含维度 |
|--------|--------|---------|
| `function_tree` | 功能树 | general、workflow_orchestration、rag_knowledge_base、model_support、enterprise_features |
| `pricing_model` | 定价模型 | free_tier、starting_price、enterprise_pricing、cost_structure、deployment_options |
| `user_persona` | 用户画像 | target_users、use_cases、technical_complexity、integration_requirements |

每个证据片段（evidence_item）和每个断言（claim）都必须归属于某个 schema_key。

### 5.3 Run（分析任务）

一个 **Run = 一次完整的竞品分析任务**。

创建 Run 时传入：
```json
{
  "task_brief": {
    "products": [
      {"product_name": "Dify"},
      {"product_name": "Coze"},
      {"product_name": "FastGPT"}
    ],
    "schema_type": "ai_agent_platform",
    "research_plan": { ... }
  },
  "mode": "real_time"
}
```

**三种执行模式**（`mode` 参数）：
| 模式 | 说明 | 适用场景 |
|------|------|---------|
| `real_time` | 真实采集（Collector Agent 执行真实抓取） | 演示实际采集能力 |
| `cached` | 从已有数据运行（跳过长时采集） | 快速迭代报告 |
| `replay` | 回放模式（与 cached 相同） | 演示用 |

### 5.4 Project（项目）

Project 是 Run 的容器，每个 Project 包含多个 Run（不同参数/时间点的分析）。

典型流程：
1. 创建 Project
2. 在 Project 内创建多个 Run（每次分析是独立的 Run）
3. 对比不同 Run 的报告质量

### 5.5 Rework（返工）机制

当 Reviewer Agent 发现证据不足或 Claim 质量问题时：

```
Reviewer 识别问题 → 生成 ReworkRequest
    → 【自动】Coverage Gap Rework（针对 evidence 不足的产品）
    → 【人工】Human Intervention（需要用户补充 URL 或确认）
```

**Coverage Gap Rework 流程**：
1. 报告状态为 `reviewed_with_gaps`（存在证据缺口）
2. 用户在前端 Review Center 点击"为所有缺口产品创建补证返工任务"
3. 系统生成 Rework Task，每个不达标产品一个
4. 用户输入补充 Seed URLs，点击"Execute real rework"
5. 系统执行：`collect_sources` → `extract_evidence` → `extract_facts` → 报告重新生成

### 5.6 Report（报告）

报告通过 **Deep Report v2 系统**（`deep_report.py`，7102 行）生成，包含：

- **14 个结构化章节**：执行摘要、分析范围、竞品选择逻辑、市场定位、竞品画像、对比矩阵、定价分析、生态信号、SWOT、场景建议、风险说明、评分卡、POC 计划、TCO 模型
- **证据附录**：所有引用来源的完整列表（可折叠卡片形式）
- **可信度摘要**：报告的质量元数据（字数、claims 数量、覆盖率等）
- **Markdown + HTML 两种输出格式**

报告版本管理：每次重新生成报告时创建新版本文件 `report_{run_id}_v{N}.md/json/html`，API 自动返回最新版本。

## 六、工作流（DAG）详解

### 6.1 完整节点序列（16 个节点）

```
build_task_brief          解析 task_brief，填充 WorkflowState
        ↓
plan_schema               生成 Schema 分析计划
        ↓
plan_sources              生成来源采集计划
        ↓
collect_sources           【采集节点】并行抓取 URL（3 级 fallback）
        ↓
evaluate_evidence         证据质量评分 + Evidence Contract 门控
        ↓
pii_scrub                PII 脱敏（目前为透传）
        ↓
extract_facts             从证据中提取结构化事实
        ↓
detect_schema_gaps        检测哪些 Schema 维度缺乏证据
        ↓
coverage_critic           覆盖率批判分析
        ↓
analyze_dimensions        Analyst Agent 生成 Claims
        ↓
review_claims             Reviewer Agent 签署 Claims，生成 Rework Requests
        ↓
reflect_on_review         审核反思（决定是否进入返工）
        ↓
prepare_human_intervention 准备人工干预点
        ↓
execute_rework            执行返工任务（自动或人工触发）
        ↓
write_report_v2           【核心】Deep Report v2 生成章节化报告
        ↓
final_review             对完整报告进行质检
        ↓
export_report            确定报告最终状态（draft / exported / blocked）
        ↓
compute_metrics          计算 8 项质量指标
```

### 6.2 节点超时配置

| 节点 | 超时 | 说明 |
|------|------|------|
| `collect_sources` | 30 分钟 | 采集容易超时，已实现文件级 checkpoint 保护 |
| `write_report_v2` | 30 分钟 | LLM 生成章节可能耗时较长 |
| 其他节点 | 5 分钟 | 一般在 30s~3min 内完成 |

### 6.3 迭代保护

- Claims Rework：最多 3 轮
- Coverage Rework：最多 3 轮
- 单节点最大递归：10007 层（LangGraph 限制）

### 6.4 条件路由

- `review_claims` 之后：有 `rework_required_claims` → `execute_rework`，否则 → `reflect_on_review`
- `reflect_on_review` 之后：`execute_rework` 完成后 → 回到 `analyze_dimensions`，否则 → `prepare_human_intervention`
- `final_review` 之后：存在 UNSUPPORTED_REPORT_SPAN → `write_report_v2`，否则 → `export_report`

**推荐的工作流**：
```bash
# 1. 创建 Project
curl -X POST http://localhost:8000/api/projects \
  -H "Content-Type: application/json" \
  -d '{"project_name": "Dify vs Coze 分析"}'

# 2. 在 Project 内创建 Run
curl -X POST http://localhost:8000/api/projects/{project_id}/runs \
  -H "Content-Type: application/json" \
  -d '{"task_brief": {...}, "mode": "real_time"}'

# 3. 异步启动
curl -X POST http://localhost:8000/api/runs/{run_id}/start-async

# 4. 轮询直到完成（每 2 秒）
while true; do
  status=$(curl -s http://localhost:8000/api/runs/{run_id}/live | python3 -c "import sys,json; print(json.load(sys.stdin)['status'])")
  echo "Status: $status"
  [ "$status" = "completed" ] && break
  sleep 2
done

# 5. 获取报告
curl http://localhost:8000/api/runs/{run_id}/report/html
```

## 九、报告生成详解（重点）

### 9.1 如何触发报告生成

报告在 workflow 的 `write_report_v2` 节点自动生成。如果只想单独生成报告（不用跑完整 workflow），可以用 E2E 测试脚本：

```bash
# 直接生成报告（从已有 evidence/claims/facts）
cd /home/shijialin/paperworking/workflow_new/productinsight-agent
source ~/anaconda3/bin/activate py10

python3 -c "
import sys, json, time
sys.path.insert(0, '.')
from backend.app.storage.db import get_connection
from backend.app.services.deep_report import run_deep_report_workflow

run_id = 'run_d56a4f411ba14c08'  # 替换为实际 run_id

with get_connection() as conn:
    evidence_rows = conn.execute('SELECT * FROM evidence_items WHERE run_id = ?', (run_id,)).fetchall()
    evidence_items = [dict(r) for r in evidence_rows]
    claim_rows = conn.execute('SELECT * FROM claims WHERE run_id = ?', (run_id,)).fetchall()
    claims = [dict(r) for r in claim_rows]
    fact_rows = conn.execute('SELECT * FROM facts WHERE run_id = ?', (run_id,)).fetchall()
    facts = [dict(r) for r in fact_rows]
    state_row = conn.execute('SELECT task_brief_json FROM runs WHERE run_id = ?', (run_id,)).fetchone()
    task_brief = json.loads(state_row[0]) if state_row else {}

products = [p.get('product_name','').strip() for p in task_brief.get('products',[]) if p.get('product_name','').strip()]
analyst_signed = [c for c in claims if c.get('review_status') == 'analyst_signed']
reviewer_signed = [c for c in claims if c.get('review_status') == 'signed']

result = run_deep_report_workflow(
    run_id=run_id,
    report_id=f'report_{run_id}_v{N}',
    products=products,
    signed_claims=reviewer_signed,
    analyst_signed_claims=analyst_signed,
    rework_required_claims=[],
    facts=facts,
    evidence_items=evidence_items,
    research_plan=task_brief.get('research_plan'),
    schema_type=task_brief.get('schema_type'),
)
print('Done, version:', result.get('report_version'))
"
```

**注意**：生成一份完整报告需要 10~20 分钟（LLM 生成 14 个章节），请耐心等待。

### 9.2 报告文件位置

每次生成报告会创建三个文件：

```
data/reports/report_{run_id}_v{N}.md    ← Markdown 源文件（主要输出）
data/reports/report_{run_id}_v{N}.json   ← 结构化 JSON（前端 API 使用）
data/reports/report_{run_id}_v{N}.html   ← HTML 渲染版本（iframe 展示）
```

其中 `N` 是版本号，从 v2 开始递增。API `/api/runs/{run_id}/report/html` **自动返回最高版本**。

### 9.3 报告可信度摘要

报告顶部有一个质量摘要，显示：

| 指标 | 含义 |
|------|------|
| 总字数 | 报告总字符数（目标 ≥7000） |
| 章节数 | 主体章节数（14 个固定 + 附录） |
| Signed Claims | Reviewer 正式签署的断言数量 |
| Reviewer 签署 | 经过 Reviewer 质检的 Claims |
| Analyst 预签 | Analyst 预先签署但未经过 Reviewer 复核的 Claims |
| Rework Required | 需要返工的 Claims |
| 平均深度 | 各维度证据充分程度的综合评分 |

### 9.4 报告状态

| 状态 | 含义 |
|------|------|
| `reviewed_with_gaps` | 报告已完成，但部分维度存在证据缺口（正常状态） |
| `reviewed` | 全部通过，无缺口 |
| `blocked` | 证据严重不足，报告被阻断（降级为预评估报告） |
| `draft` | 草稿状态（正在生成中） |

**重要**：即使状态是 `blocked`，系统也会生成一份"预评估报告"，内容仍然可用，只是会标注"预评估阶段"警告。**系统永远不会输出 failed 状态**——任何错误都会优雅降级。

### 10.4 Evidence Sufficiency Sprint——解决"永远证据不足"（2026-06-03）

**根因**：6 个叠加 bug 导致系统永远无法通过 evidence gate，最终只能生成 blocked 预评估报告。

#### Bug A: 中文 evidence 评分误杀
**症状**：中文来源的 evidence 全部 `usable_for_claim=False`，得分仅 0.1
**根因**：`_calc_information_density` 和 `_calc_schema_fit` 的 feature words 列表只有英文
**修复**（`evidence_evaluator.py`）：
- `SCHEMA_KEYWORDS` + `DIMENSION_KEYWORDS` 加中文关键词
- `_calc_schema_fit` 加中文 regex
- `is_meaningful_evidence` 对中文用字符数判断
- `_calc_relevance` schema key step 对中文加 substring 匹配
- `_schema_keywords_lower` 不再过滤中文

#### Bug B: Schema 映射错误
**症状**：业务价值内容被归入 pricing 再被 gate 拒绝
**根因**：Schema keywords 中 `"cost"` 和 `"enterprise"` 同时属于多个维度，导致错误归类
**修复**（`evidence_extractor.py`）：
- `pricing_model` keywords 去掉 "cost"/"enterprise"
- 去失落 standalone `function_tree` 条目
- 加 `business_value` schema key
- 所有维度加中文关键词

#### Bug C: Stale Evidence ID
**症状**：`review_claims` 过滤后的 evidence_ids 没有写回 `state["claim_drafts"]`
**根因**：`filtered_evidence_ids` 仅用于后续处理，但从未回写 state
**修复**（`nodes.py`）：
- `review_claims` 新增 `_has_usable_evidence` hard gate（无 usable evidence 的 claim 直接降级）
- `updated_claims` 写回 `state["claim_drafts"]`

#### Bug D: `_default` gate 误杀
**症状**：所有无明确规则的维度，`web_page` 来源全部被拒绝
**根因**：`elif reqs.get("allowed")` 对空列表 `[]` 返回 truthy，导致拒绝
**修复**（`deep_report.py`）：`elif` 改为 `if reqs.get("allowed") is not None`，空列表不再触发拒绝

#### Bug E: `_run_id` 为空导致 coverage 全零
**症状**：E2E 测试的 claims 无 `run_id` 字段，导致 product slug extraction 失败，coverage 全部为 0
**根因**：`_build_render_context` 从 `signed_claims[0].run_id` 提取，但测试数据无此字段
**修复**（`deep_report.py` + `nodes.py`）：`_build_render_context` 增加 `run_id` 参数兜底；claim evidence re-evaluation 时补充 `run_id`

#### Bug F: Evidence re-evaluation 缺失
**症状**：旧 DB 里的 evidence 用旧规则评分（0.1），新规则不生效
**修复**（`test_e2e_full.py`）：重新评估证据（`evaluate_evidence_items`）；remap claim evidence_ids

### 10.9 报告查看器（Report Viewer）Markdown 内容回退修复（2026-06-09 下午）

**问题**：报告查看器前端（`report_viewer.py`）始终渲染旧内容，即使后端已生成新的 `report_*_v2.md` 文件。

**根因排查过程**：

1. Streamlit 重启正常（16:59、17:13、17:25 三次），代码已更新
2. 浏览器硬刷新、隐私模式均无效
3. API `/api/runs/{run_id}/report/json` 返回的 sections 并不为空——每个章节有标题（10~790词），但这些内容全部是**后端 JSON 占位数据**，只有标题没有正文
4. 原有 `empty_sections` 判断条件：`all(not sec.get("content_markdown", "").strip() for sec in sections)` → 对 JSON 数据永远为 False（因为每个 section 都有至少一个标题词），**导致 markdown fallback 永远不触发**
5. 后端 markdown 文件（87KB、4888词）包含完整的报告正文，但前端从未读取它

**修复方案**：采用"内容丰富度对比"策略，替代原有的"空检查"策略。

**修改文件**：`productinsight-agent/frontend/views/report_viewer.py`

修改 1 — `render_report_viewer()` 中的 fallback 逻辑：

```python
# 修改前（永远不触发）
empty_sections = all(
    not sec.get("content_markdown", "").strip()
    for sec in sections
)
if empty_sections:
    # ... fetch markdown ...

# 修改后（内容丰富度对比）
sections = data.get("sections", [])
json_total_words = sum(len(sec.get("content_markdown", "").split()) for sec in sections)
markdown_used = False

try:
    md_resp = _req.get(f"{API_BASE}/api/runs/{run_id}/report/md", timeout=15)
    if md_resp.status_code == 200:
        md_text = md_resp.text
        md_sections = _parse_sections_from_markdown(md_text)
        md_total_words = sum(len(sec.get("content_markdown", "").split()) for sec in md_sections)
        # 当 markdown 内容量 ≥ JSON 的 120% 时，优先使用 markdown
        if md_sections and (md_total_words >= json_total_words * 1.2 or md_total_words > json_total_words + 500):
            sections = md_sections
            markdown_used = True
            st.info(f"📄 Markdown优先: JSON({json_total_words:,}词) → Markdown({md_total_words:,}词, {len(md_sections)}章节)")
```

修改 2 — `_parse_sections_from_markdown()` 的解析逻辑：

```python
# 修改前（跳过所有编号章节，导致正文内容丢失）
if re.match(r'^\d+\.', title):
    continue  # ← 错误：把所有编号章节都跳过了

# 修改后（保留命名章节 + 合并编号章节）
if re.match(r'^\d+\.', title):
    numbered_blocks.append((title, body))  # 收集，后续合并
else:
    named_sections.append({...})           # 命名章节直接保留

# 编号章节合并为单个"📑 报告正文"块（按顺序排列，去除编号前缀）
if numbered_blocks:
    merged_parts = []
    for num_title, body in numbered_blocks:
        display_title = re.sub(r'^\d+\.\s*', '', num_title)  # "13. 场景化选型建议" → "场景化选型建议"
        merged_parts.append(f"### {display_title}\n\n{body}")
    report_body_text = "\n\n---\n\n".join(merged_parts)
    named_sections.append({"section_title": "📑 报告正文", "content_markdown": report_body_text, ...})
```

| 章节数 | 17 | 8 |
| 总词数 | 2,112 | 4,888 |
| 产品概览卡片 | 0词（占位符） | 226词（完整内容） |
