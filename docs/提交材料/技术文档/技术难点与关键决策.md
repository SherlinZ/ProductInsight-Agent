# 技术难点与关键决策

> **说明**：本节整理项目开发过程中的主要技术难点、方案选择依据及关键权衡，供评委和技术同行参考。

---

## 一、工作流编排方案的选择

比赛课题建议使用 LangGraph 或 CrewAI 进行多 Agent 编排。

在开发初期，项目确实尝试了 LangGraph 的 StateGraph 机制，但在集成过程中发现了一个状态传播问题：`review_claims` 节点执行后将审查结果写入 `state["signed_claims"]`，但在后续 `write_report` 节点中，读取到的 `signed_claims` 仍为空列表。经排查，该问题为 LangGraph 内部的状态传播机制缺陷。

考虑到 Demo 演示对稳定性的要求，项目改用确定性顺序执行方案（`force_sequential=True`）：每个工作流节点执行后将结果写入 `WorkflowRepository`，后续节点直接从数据库读取。这一方案在保证执行稳定的同时，仍然完整保留了以下能力：

- 每个节点的执行状态记录
- 从任意节点断点重跑
- LLM 调用级别的 Trace 记录
- 前端 DAG 进度可视化

核心实现位于 `backend/app/orchestrator/nodes.py`（5821行），工作流定义位于 `backend/app/orchestrator/graph.py`（1001行）。

## 二、幻觉抑制的多层机制

多 Agent 协作系统的可信度很大程度上取决于能否有效抑制大模型幻觉。项目在架构层面实现了三层防护机制：

**第一层：引用强制。** 报告正文中每条 `[E:N]` 引用编号必须在 Evidence Appendix 中存在对应条目。系统在报告导出前执行引用有效性校验，引用编号缺失或越界均会触发导出阻断。

**第二层：Evidence Contract 门控。** 每条证据在进入 Claims 生成前必须通过 6 维度量化评分（相关性、权威性、时效性、维度对齐度、信息密度、综合分），综合分低于 0.32 的证据被标记为 `usable_for_claim=False`，无法支撑任何分析结论。同时，每个分析维度设有来源类型白名单（如 `pricing_model` 维度仅接受官方定价页来源，拒绝第三方博客和社交媒体）。

**第三层：Reviewer 硬质检。** Reviewer Agent 对每条 Claim 执行 8 项检查，任何一项失败均会触发 `rework_required` 状态，并将审查结果精确映射到对应 Agent/节点，形成闭环打回。

此外，系统设有 `blocked` 降级机制：当 Evidence 质量不满足合约要求时，Writer Agent 进入受限状态，禁止生成强推荐语言，报告状态和评分卡均做降级处理。

核心实现：`backend/app/services/evidence_evaluator.py`（Evidence 评分）、`backend/app/agents/reviewer/reviewer.py`（质检体系）、`backend/app/agents/writer/writer.py`（blocked 机制）。

## 三、Evidence 质量评分的调优过程

Evidence 质量评分体系并非一次性设计完成，而是经历了多轮迭代调优：

初始版本的 `FINAL_SCORE_THRESHOLD` 设为 0.45，发现对于短文本证据（如定价引用，通常 50-150 字符）过于严格，导致大量有效证据被错误过滤。在 Evidence-Sufficiency Sprint 中，经 E2E 实测数据验证后将阈值逐步下调至 0.32。

在支持中文产品来源（Dify、Coze、FastGPT 等）时，发现产品名称的变体形式（如 Coze 的中文名"扣子"）导致相关性计算偏低，相应将 `RELEVANCE_THRESHOLD` 从 0.30 调整为 0.20，同时对中文内容应用语义密度加成。

此外，系统建立了 30+ 条导航噪声黑名单（GitHub 目录页、404 页面、Cookie 弹窗等）和官方产品域名注册表（排除 dify-china.com 等第三方镜像站），确保评分体系不受劣质来源干扰。

## 四、Schema 体系的动态适配

项目设计了 6 种分析 Schema Type，对应不同的分析场景：

| Schema Type | 主要维度 | 适用场景 |
|------------|---------|---------|
| ai_agent_platform | 功能树、定价、企业就绪度等6项 | AI Agent 平台竞品分析 |
| competitor_landscape | 功能树、定价、用户画像等6项 | 宽泛竞品概览 |
| pricing_analysis | 定价模型、价值主张、TCO 等6项 | 定价深度对比 |
| knowledge_management | 知识结构、协作体验、AI 辅助等9项 | 知识管理工具分析 |
| ai_coding_assistant | 编程能力、IDE 集成、安全等6项 | AI 编程工具对比 |
| sales_battlecard | 优势、劣势、差异化3项 | 销售话术卡 |

系统根据用户输入的查询内容自动推断应使用的 Schema 类型（`infer_schema_type` 函数），并加载对应的维度集合、来源白名单和报告章节模板。不同 Schema 的输出遵循统一的 Claim 格式，保证报告结构的一致性。
