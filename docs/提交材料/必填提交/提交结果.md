## 二、功能说明

### 核心功能清单

1. **Research Plan 自动生成与人工确认**  
   用户输入自然语言竞品分析需求后，系统自动生成研究计划，包括竞品对象、分析维度、信息来源计划、执行节点和报告大纲。用户可以在执行前审核与调整方案，避免系统直接进入不可控的自动分析流程。

2. **多角色 Agent 协作分析**  
   系统包含 Collector、Analyst、Reviewer、Writer 等角色 Agent。Collector 负责公开来源采集与证据整理，Analyst 负责生成结构化分析 claims，Reviewer 负责审查证据支撑与可信度，Writer 负责基于审核结果生成竞品分析报告。

3. **Evidence → Claim → Review → Report 可信生成链路**  
   系统不是直接让大模型生成报告，而是先从公开来源形成 evidence，再基于 evidence 生成 claims，并由 Reviewer 审查后进入报告。证据不足的内容会标记为 evidence gap 或预评估状态，避免无依据强结论。

4. **17 步可观测工作流**  
   系统将一次竞品分析拆解为 17 步可追踪工作流，覆盖任务简报、schema 规划、source plan、信息采集、evidence 抽取、PII 检查、facts 抽取、coverage 评估、claims 生成、Reviewer 审查、人工介入、报告撰写、终审、导出与指标计算等节点。

5. **报告自动生成与多格式交付**  
   系统支持生成 Markdown、HTML 和 JSON 三种报告产物。报告内容包括管理层摘要、竞品概览、功能与部署对比、定价与商业模式分析、SWOT / 风险分析、场景化建议、证据附录和可信度说明。

6. **Trace / Audit 全链路可观测**
   系统记录 Agent 节点状态、LLM 调用、工具执行、输入输出、中间产物和报告导出结果。用户可以在前端查看工作流进度、节点状态、报告质量指标和 Trace 信息，支持过程复盘与答辩展示。

7. **章节并行生成与多轮修订**
   `write_report_v2` 内部采用多线程并行生成多个章节（3 个章节同时写），每个章节内部还有最多 3 轮"生成→Reviewer 审核→打回修订"的迭代闭环，形成 Writer Agent 与 Reviewer Agent 之间的实时协作与质量把控。

8. **Evidence Contract 硬门控**
   系统对证据来源类型设置了维度白名单：例如 `pricing_model` 维度只接受官方来源，拒绝第三方文章；`security/compliance` 维度同样拒绝 blog 和社交媒体。每条 evidence 在进入 Claim 生成前必须先通过 Evidence Contract 门控，从来源侧杜绝劣质证据支撑关键断言。

9. **采集层 3 级 Fallback 降级**
   数据采集模块（`web_fetcher.py`）实现 3 级降级策略：① 直接 `requests` 抓取；② 失败则启用 Playwright 浏览器自动化抓取；③ 还失败则用 Doubao Search API 搜索兜底。同时配合 `multi_round_search_service.py` 实现多轮搜索补证链路，确保采集稳定性。

### 端到端使用流程

1. 用户进入前端 `Analysis Flow` 页面，输入自然语言竞品分析需求，例如“对比 Dify、Coze 和 FastGPT 在企业 AI Agent / 低代码 AI 应用平台方向的能力”。

2. 系统调用 Research Planner 生成 Research Plan，包括竞品列表、分析目标、分析维度、来源计划、执行工作流和报告大纲。

3. 用户在 Plan Review 阶段查看并确认研究计划，可根据需要调整竞品对象、分析维度或报告结构。

4. 用户点击“开始分析”后，系统创建 Project 和 Run，并触发 17 步可观测工作流。

5. 工作流按阶段执行信息采集、证据抽取、事实整理、coverage 检查、claims 生成、Reviewer 审查、报告撰写和终审导出。

6. 若证据不足或来源类型不满足要求，系统会标记 evidence gap、触发人工介入或以预评估方式呈现，避免强行生成未经支撑的确定性结论。

7. 工作流完成后，用户可在 Deliverables / Report Viewer 页面查看最终报告，包括对比矩阵、SWOT 分析、图表、证据附录和可信度说明。

8. 用户可在 Trace / Audit 页面查看 Agent 执行过程、节点输入输出、LLM 调用和关键中间产物，实现报告生成过程的可复盘与可审计。

### 大模型 / AI 能力使用说明

#### 模型调用

- **主模型**：Doubao-Seed-2.0-lite（火山引擎方舟 API）
- **调用方式**：REST API（`https://ark.cn-beijing.volces.com/api/v3/chat/completions`）
- **用途**：Research Plan 生成、Evidence 抽取、Claims 分析、Claims 质检（review）、报告撰写（section write + review）、对比表生成、SWOT 图表

#### Agent 设计

| Agent | 职责 | 主要输入 | 主要输出 |
| --- | --- | --- | --- |
| CollectorAgent | 采集竞品官网、文档、公开资料并生成来源记录 | source plan、seed URLs、搜索结果 | sources、snapshots、raw content |
| Evidence Extractor | 从页面或文本中抽取结构化证据 | raw content、schema | evidence items |
| Evidence Evaluator | 对每条 evidence 进行 6 维度质量评分（相关性/权威性/时效性/维度对齐度/信息密度/纯净度），只有综合分 >= 0.5 且 `usable_for_claim=True` 才放行；并执行 Evidence Contract 门控（来源类型白名单校验） | evidence items | graded evidence、gate_rejection 标记 |
| AnalystAgent | 基于 evidence 生成结构化分析 claims | evidence、facts、analysis dimensions | claim drafts |
| ReviewerAgent | 审查 claims 的证据支撑、来源类型、schema 对齐和可信度 | claim drafts、evidence items | signed claims、evidence gaps、rework flags |
| WriterAgent | 基于 signed / pre-signed claims 组织报告章节 | claims、report outline、evidence appendix | Markdown / HTML / JSON report |
| HumanReview Node | 在高风险或证据不足时提供人工确认入口 | review issues、coverage gaps | approve / reject / request rework |

#### 反馈机制

- ReviewerAgent 标记 `rework_required=True` 时，自动触发 `reflect_on_review` → `execute_rework` 循环（最多 3 轮）
- coverage_critic 评估 evidence gaps，若关键维度无证据则打回 `execute_rework`
- 无效 claims 被标记为 `evidence_gap`，报告相应章节注明证据缺口
- **两套 Rework 回路**：
  - **Coverage Gap Rework**：evidence 不足 → 打回 `plan_sources` → 重新采集 → 重新评估（最多 3 轮）
  - **Claim Rework**：Reviewer 打回 → `analyze_dimensions` 重新生成 Claims → 重新 Review（最多 3 轮）
  - 两套回路互不影响，可独立触发，形成完整的多轮迭代反馈闭环

#### AI 与确定性规则的分工

系统并未将所有判断都交给大模型，而是采用“LLM 生成 + 规则校验 + Reviewer 门控”的组合方式：

- LLM 负责需求理解、分析生成、报告撰写和解释性内容生成；
- 确定性代码负责 schema 校验、字段完整性检查、证据绑定、引用编号校验、PII 处理、状态流转和导出控制；
- Reviewer Agent 负责判断 claims 是否有充分 evidence 支撑，并在证据不足时触发 evidence gap 或预评估状态。

这种设计能够降低单轮 Prompt 直接生成报告带来的幻觉风险，也更符合企业竞品分析对可信度、可审计性和可复核性的要求。

#### blocked 降级机制

当 evidence 质量不满足 Evidence Contract、或存在 `rework_required` claim 未解决时，Writer Agent 进入 `is_blocked` 模式：报告禁止任何"最优"/"top pick"/"🥇"等强推荐语言；报告状态置为 `blocked_consistency`；评分卡降级为"暂无法评分"，所有产品标注"暂不参与排名"；POC 验证清单和 TCO 成本模型均降级为"待核验"；Evidence Appendix 仍正常输出，标注每条 evidence 的来源与 Reviewer 状态。

### 关键工程难点与解决方案

| 难点 | 解决方案 |
| --- | --- |
| 长流程状态传递与演示稳定性 | 将复杂竞品分析拆解为 17 步可观测工作流。当前 Demo 采用确定性顺序调度执行节点，每步结果写入 WorkflowRepository 和 TraceRepository，保证演示稳定、可复现、可回放 |
| Agent 输出格式不稳定 | 通过 Pydantic Schema、字段白名单、analysis dimensions 约束和 Reviewer 审核机制保证输出结构一致；不符合要求的 claim 标记为 evidence gap 或预评估内容 |
| 大模型幻觉风险 | 采用 Evidence → Claim → Review → Report 链路，结合引用强制、证据来源检查、定价 sanitizer、内部一致性 gate 和报告导出前检查，减少无证据结论 |
| 信息采集不稳定 | 支持 seed URL、搜索发现和 replay demo 数据多种方式；正式演示可使用稳定 Demo 数据，避免现场网络波动、目标网站阻断或实时爬取超时影响展示 |
| 报告空泛或缺少业务价值 | 使用 report outline、section writer、对比矩阵、SWOT / 风险分析、场景化建议和证据附录构成完整报告结构，使输出更接近企业产品团队可读的竞品分析材料 |
| 可观测与答辩复盘 | 对每个节点记录状态、输入输出、LLM 调用、错误信息和中间产物；前端提供 Running Center、Trace / Audit、Deliverables 页面支持展示和复盘 |

## 五、结果说明

### 5.1 项目完成度

**当前状态：端到端可演示 Demo，核心竞品分析闭环已贯通。**

ProductInsight Agent 当前已完成从“用户输入竞品分析需求”到“生成结构化竞品分析报告”的端到端流程，具备比赛课题要求的多 Agent 协作、结构化知识抽取、DAG 化任务流转、证据溯源、质检审核、报告生成与前端展示能力。当前版本已完成本地环境和服务器环境验证，可通过演示视频展示完整业务流程，并可根据 README 在本地复现运行。

当前系统已实现的核心能力包括：

1. **需求输入与 Research Plan 生成**  
   用户可以在前端输入自然语言竞品分析需求，系统自动生成 Research Plan，包括竞品对象、分析维度、信息来源计划、执行工作流和报告大纲。用户可在执行前审核和调整计划，避免系统直接进入不可控的自动生成流程。

2. **多 Agent 协作执行链路**  
   系统将竞品分析过程拆解为采集、证据抽取、事实整理、竞品分析、质检审核、报告撰写和最终导出等多个阶段。Collector、Analyst、Reviewer、Writer 等 Agent 按职责协作，避免单个大模型直接生成报告导致的不可控和不可追溯问题。

3. **17 步可观测工作流**  
   系统采用 DAG 化任务建模，将一次竞品分析拆解为 17 个可追踪执行节点，覆盖 task brief 构建、schema 规划、source plan、信息采集、evidence 抽取、PII 检查、facts 抽取、coverage 评估、claims 生成、Reviewer 审查、人工介入、报告撰写、终审、导出和指标计算等步骤。每个节点均记录执行状态、中间产物和 Trace 信息，便于演示、复盘和问题定位。

4. **Evidence → Claim → Review → Report 可信链路**  
   系统不是直接让大模型生成报告，而是先从公开来源抽取 evidence，再由 Analyst 生成结构化 claims，随后由 Reviewer 对 claims 的证据支撑、来源类型、schema 对齐和可信度进行审查，最终由 Writer 基于通过审核或带有预评估标记的 claims 生成报告。证据不足的内容会被标记为 evidence gap 或预评估状态，避免生成无依据的强结论。

5. **报告产物生成与展示**  
   当前版本支持生成 Markdown、HTML 和 JSON 三种形式的报告产物。代表性 Demo 已能输出结构化竞品分析报告，包含管理层摘要、竞品对比矩阵、功能与部署分析、SWOT 分析、场景化建议、风险提示、证据附录和可信度说明等内容，适合作为企业产品团队竞品分析的初步决策材料。

6. **Trace 与可观测能力**  
   系统记录 Agent 节点状态、LLM 调用、工具执行、输入输出和中间产物，可在前端查看工作流执行情况、报告生成状态和节点级 Trace。这一机制能够支撑答辩中的 Agent 决策回放和可信性说明。

7. **稳定演示与本地复现**  
   当前版本支持使用稳定 Demo 数据进行演示，避免现场网络波动、目标网站访问失败或实时采集超时影响展示效果。同时保留本地启动方式，评委或复现者可根据 README 启动 FastAPI 后端和 Streamlit 前端进行体验。

总体而言，当前项目已经达到“可演示 Demo / MVP+”状态：核心技术链路完整，主业务闭环可跑通，关键结果可展示，已满足比赛提交和答辩演示的基本要求。后续若继续迭代，可重点增强公网部署、多领域泛化、实时采集稳定性、复杂 rework 闭环和更精细的产品化交互体验。

### 5.2 代表性 Demo 结果

当前稳定演示场景为 AI Agent / 低代码 AI 应用平台竞品分析，围绕 Dify、Coze、FastGPT 等产品展开。系统在该场景中展示以下结果：

- 完成从自然语言需求到 Research Plan 的自动生成；
- 完成从 Research Plan 到 Project / Run 的创建；
- 完成 17 步可观测工作流执行；
- 生成结构化竞品分析报告；
- 输出竞品对比矩阵、SWOT / 风险分析和场景化建议；
- 保留 Evidence Appendix，用于报告结论溯源；
- 记录 Agent 执行 Trace，用于过程复盘；
- 对证据不足内容进行预评估标记或可信降级，避免强行生成未经支撑的结论。

### 5.3 项目亮点 / 创新点

#### 亮点 1：证据优先的可信生成链路

系统采用 “Evidence → Claim → Review → Report” 的生成链路，而不是直接让大模型一次性生成报告。每条关键分析结论都需要绑定 evidence，并经过 Reviewer 审查后才能进入报告。对于证据不足的结论，系统会标记 evidence gap 或以预评估方式呈现，体现 “No evidence, no claim” 的可信生成原则。

#### 亮点 2：多 Agent 协作与可观测 DAG 工作流

系统将竞品分析拆解为多个可追踪节点，由 Collector、Analyst、Reviewer、Writer 等角色 Agent 分工协作。报告生成阶段（`write_report_v2`）还实现了多线程并行机制：最多 3 个章节同时生成（ThreadPoolExecutor），每个章节内部还有最多 3 轮"生成→Reviewer 审核→打回修订"的迭代闭环，形成 Writer Agent 与 Reviewer Agent 之间的实时协作与质量把控。相比单轮 Prompt 直接生成报告，该方案更符合企业场景中对可控性、可审计性和可复核性的要求。

#### 亮点 3：Evidence Contract 与量化质检

系统不仅依赖 Reviewer Agent 的人工判断，还实现了量化的 Evidence Contract 机制：每条 evidence 在进入 Claims 生成前必须先通过 6 维度质量评分（相关性/权威性/时效性/维度对齐度/信息密度/纯净度），综合分 < 0.5 或 `usable_for_claim=False` 则直接拒绝进入报告。同时，Evidence Contract 对每个分析维度设定了来源白名单（如 `pricing_model` 维度只接受官方文档，拒绝第三方文章和社交媒体），从来源侧杜绝劣质证据支撑关键断言。

#### 亮点 4：可信阻断与 blocked 降级

当 evidence 质量不满足 Evidence Contract、或存在 `rework_required` claim 未解决时，Writer Agent 进入 `is_blocked` 模式：禁止任何"最优"/"top pick"等强推荐语言；报告状态置为 `blocked_consistency`；评分卡降级为"暂无法评分"，POC 验证清单和 TCO 模型降级为"待核验"。这使系统在不确定信息环境下保持谨慎输出，降低大模型幻觉和错误决策风险。

#### 亮点 5：Schema 驱动的结构化竞品分析

系统通过竞品知识 Schema 约束分析维度和报告结构，使不同 Agent 的输出能够围绕功能、定价、用户画像、部署方式、企业能力等维度组织，保证报告格式一致、字段可比、内容可扩展。该机制支持后续迁移到其他企业产品、SaaS 工具或行业竞品分析场景。

#### 亮点 6：两套 Rework 迭代回路

系统实现了独立运作的两套 Rework 回路：① **Coverage Gap Rework**（evidence 不足 → 打回 `plan_sources` → 重新采集 → 重新评估，最多 3 轮）；② **Claim Rework**（Reviewer 打回 → `analyze_dimensions` 重新生成 Claims → 重新 Review，最多 3 轮）。两套回路互不影响、可独立触发，形成完整的多轮迭代反馈闭环。

#### 亮点 7：采集层 3 级 Fallback 降级

数据采集模块（`web_fetcher.py`）实现 3 级降级策略：① 直接 `requests` 抓取；② 失败则启用 Playwright 浏览器自动化抓取；③ 还失败则用 Doubao Search API 搜索兜底。同时配合 `multi_round_search_service.py` 实现多轮搜索补证链路，确保演示过程不因网络波动或目标站点阻断而中断。

#### 亮点 8：工程化可复现与答辩友好

项目提供前后端分离的本地运行方式，后端基于 FastAPI，前端基于 Streamlit，数据层使用 SQLite 与 JSON 报告文件，便于快速部署和复现。系统同时准备了演示视频、技术文档、答辩幻灯片和运行说明，适合比赛评审快速理解项目能力。
