## 一、多 Agent 协作与输出可信度（35%）

### 1.1 角色划分清晰，多个专职 Agent

| 评分要点 | 实现情况 | 证据 |
|---------|---------|------|
| 采集 Agent | ✅ `CollectorAgent` — URL 发现、网页抓取、snippet 抽取 | `backend/app/agents/collector/` |
| 分析师 Agent | ✅ `AnalystAgent` — 基于 evidence 生成 Claims | `backend/app/agents/analyst/analyst.py`（735行） |
| 报告撰写 Agent | ✅ `WriterAgent` — 基于签署 Claims 生成报告 | `backend/app/agents/writer/writer.py`（848行） |
| 质检 Agent | ✅ `ReviewerAgent` — 8项检查 + Evidence Contract | `backend/app/agents/reviewer/reviewer.py`（785行） |
| 职责边界明确 | ✅ 4个 Agent 各司其职，通过 `WorkflowState` TypedDict 传递数据 | 详见架构图 |

### 1.2 编排框架使用合理

| 评分要点 | 实现情况 | 证据 |
|---------|---------|------|
| LangGraph / CrewAI | ⚠️ 代码中用确定性顺序调度（`force_sequential=True`）而非 LangGraph 状态机 | `backend/app/orchestrator/graph.py` |
| DAG 任务流转可视化 | ✅ 17步工作流，每步状态写入 `WorkflowRepository` | `backend/app/orchestrator/nodes.py`（5821行） |
| 可追溯 | ✅ `TraceRepository` 记录每步 LLM 调用 | `backend/app/tracing/llm_trace.py` |

> **说明**：LangGraph 的 state 传播存在 bug，导致 `review_claims` 修改的 signed_claims 后续节点读取不到。改用确定性顺序 Python 执行（每步结果写 DB，下一步直接读取）作为稳定复现方案，同时保留了节点状态追踪和断点重跑能力。

### 1.3 Agent 间结构化消息传递

| 评分要点 | 实现情况 | 证据 |
|---------|---------|------|
| Function Calling | ✅ 各 Agent 输出为 Pydantic Schema 结构化 JSON，非纯自然语言对话 | 各 Agent 的 `analyze()`/`review_claim()`/`write()` 方法 |
| 标准 Schema | ✅ `WorkflowState` TypedDict 作为消息总线，结构化传递 | `backend/app/orchestrator/state.py` |
| 非纯对话 | ✅ Agent 间通过 `state["claim_drafts"]` / `state["signed_claims"]` 等结构化字段通信 | `nodes.py` 17个节点 |

### 1.4 反馈闭环真实可触发

| 评分要点 | 实现情况 | 证据 |
|---------|---------|------|
| 质检打回采集/分析 Agent | ✅ `ReviewerAgent.review_claim()` → 8种 reason_code → 精确映射到目标 Agent/节点 | `reviewer.py` 第50-70行 |
| 重做后输出有改善 | ✅ Coverage Gap Rework：证据不足 → 重新采集 → 重新评估，Before/After 数字写入 DB | `0608存在问题.md` M4 节 |

| **两套独立 Rework 回路** | ✅ Coverage Gap Rework + Claim Rework，互不影响、可独立触发 | `提交结果.md` 亮点6 |

### 1.5 Schema 驱动的结构化输出

| 评分要点 | 实现情况 | 证据 |

## 总结：项目自评

### 强项（✅）

1. **多 Agent 协作链路完整**：4个专职 Agent，17步 DAG，结构化通信，非纯对话
2. **可信生成机制扎实**：Evidence → Claim → Review → Report 四链路 + Evidence Contract 硬门控 + blocked 降级
3. **Schema 体系完整**：6种分析类型自动推断，动态切换，跨行业可扩展
4. **量化质检体系**：6维度 Evidence 评分 + 8项 Claim 质检 + 原因码精确打回
5. **工程实现深度**：5821行 nodes.py、1899行 research_planner.py、830行 evidence_evaluator.py，远超 Demo 玩具级别

### 待强化（⚠️）

1. **LangGraph 编排**：用确定性顺序替代 LangGraph 状态机，需在答辩时解释原因（LangGraph state 传播 bug）
2. **效率量化数据**：传统 vs 系统的时间对比，评委问"提升了多少"时需要有数字
3. **Git/TRAE 痕迹**：如有要求需补充
4. **演示视频**：待录制
