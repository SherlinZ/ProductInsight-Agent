# ProductInsight Agent — 场景落地设想

## 一、目标客户画像

### 1.1 企业产品团队

**典型场景**：企业产品经理（PM）需要定期完成竞品分析报告，用于：
- 新产品立项前的市场调研
- 现有产品迭代方向的决策参考
- 对标竞品的定价和功能策略调整

**痛点**：
- 每次分析都需要人工搜集大量公开资料，耗时 1-3 天
- 团队成员各自搜集，信息分散，难以形成统一结构
- 最终报告质量依赖个人经验，缺乏可复用性

**ProductInsight Agent 的价值**：
- 将 1-3 天的人工调研压缩到 30 分钟以内的系统执行 + 人工审核
- 所有分析结论附带证据来源，支持团队复核和讨论
- 标准化输出格式，便于积累和对比历史分析

### 1.2 投资机构 / 行业研究团队

**典型场景**：投资经理或行业研究员需要快速了解某个赛道（如 AI Agent 平台、低代码工具）的竞品格局

**痛点**：
- 信息源分散，需要覆盖官网、文档、社区、财报等多个渠道
- 需要快速生成结构化对比材料用于内部分享

**ProductInsight Agent 的价值**：
- 自动抓取多来源信息，生成带溯源的结构化报告
- 可快速切换竞品对象，分析维度保持一致

### 1.3 SaaS / 企业软件销售团队

**典型场景**：售前工程师在客户交流前，需要快速了解客户当前使用的竞品情况

**痛点**：
- 需要在短时间内掌握竞品的优劣势，用于对比自家产品
- 每次都需要临时搜集资料，信息不系统

## 二、目标市场与落地路径

### 2.1 近期落地（1-3 个月）

**目标**：在企业内部（字节跳动 CIS 部门）落地试用

**路径**：
1. 选择 1-2 个业务线（如 toB SaaS 产品团队）作为试点
2. 使用内部服务器部署，系统运行在公司内网环境
3. 分析师 PM 试用并反馈，系统根据反馈迭代
4. 验证指标：分析报告生成时间 vs 人工时间、报告引用完整率

**关键成功指标（KPI）**：
| 指标 | 当前（人工） | 目标（系统） |
|------|------------|-------------|
| 单次分析耗时 | 1-3 天 | 30 分钟系统执行 + 1 小时人工审核 |
| 引用溯源覆盖率 | ~30%（靠记忆手动标注） | >80%（系统自动绑定） |
| 报告版本一致性 | 依赖个人风格 | 标准化结构，可横向对比 |

### 2.2 中期落地（3-6 个月）

**目标**：扩展到多个业务线，成为企业级竞品分析基础设施

**路径**：
1. 完善多租户和权限管理，支持不同业务线独立分析
2. 增加私有化部署能力，支持客户环境交付
3. 接入企业内部知识库（如 Confluence、内部文档），增强证据来源
4. 支持分析报告的团队协作评论和修正

**关键成功指标（KPI）**：
| 指标 | 目标 |
|------|------|
| 日活跃分析任务数 | >20 次/天 |
| 用户满意度（NPS） | >40 |
| 报告复用率（同一产品二次分析） | >50% |

### 2.3 长期愿景（6-12 个月）

**目标**：面向外部中小企业（SMB）和 SaaS 公司提供产品化服务

**路径**：
1. 标准化为 SaaS 产品，支持在线注册和使用
2. 支持自定义 Schema，企业可定义自己的分析维度体系
3. 接入更多数据源（如 G2、Capterra、AppStore 评论）
4. 增加报告导出格式（PowerPoint、PDF 定制模板）

## 三、差异化竞争优势

相比市面上已有的竞品分析方案（如 G2、TrustRadius、人工顾问），ProductInsight Agent 的差异化定位：

| 维度 | 传统方案 | ProductInsight Agent |
|------|---------|---------------------|
| **时效性** | 数据更新慢（季度/月度） | 按需实时采集 |
| **可解释性** | 结论无溯源 | 每条结论绑定原始证据 |
| **可信度** | 直接生成结论，有幻觉风险 | Evidence → Claim → Review 链路审核 |
| **灵活性** | 固定分析维度 | Schema 可自定义 |
| **成本** | 订阅制或顾问费用 | 开源可自部署 + LLM API 成本 |
| **迭代闭环** | 无反馈机制 | Coverage Gap / Claim Rework 双回路 |

## 四、当前版本的落地约束

### 4.1 已知限制

1. **采集稳定性**：部分目标站点（如 Coze 中国区）有 Cloudflare 保护，需依赖 Search API 兜底
2. **上下文窗口**：超长 evidence 列表已通过分片解决，但超长报告生成仍受 LLM context 限制
3. **中文支持**：Evidence 6 维度评分已加入中文关键词，但部分英文来源抓取质量仍有优化空间
4. **多语言分析**：当前主要针对中文信息来源场景设计

### 4.2 扩展方向

- **多语言采集**：扩展到英文官方网站和技术博客
- **私有数据源**：支持接入内部 Confluence、Slack 等私有数据
- **团队协作**：支持多用户评论、标注和修正报告
- **定时任务**：支持定期自动重新分析并对比报告版本差异
