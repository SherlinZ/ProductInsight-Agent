# ProductInsight Agent — 技术架构与项目原理

> **用途**：答辩时向评委讲解系统的内部机制、解决的问题与方法、亮点
> **读者**：评委（技术背景）、参赛者本人（演讲参考）
> **时长建议**：配合 PPT 讲解，约 6-8 分钟

---

## 一、项目要解决的核心问题

### 1.1 传统竞品分析的四个痛点

| 痛点 | 传统做法 | 后果 |
|------|---------|------|
| 信息分散 | 人工逐个访问竞品官网、文档 | 耗时数小时，信息源有限 |
| 结论难追溯 | "我觉得"、"据报道" | 评委质疑可信度 |
| 报告易幻觉 | 直接让 LLM 生成结论 | 无证据支撑，含糊或错误 |
| 复核成本高 | 人工审核，质量参差不齐 | 效率低，标准不统一 |

### 1.2 我们的解题思路

**核心理念**：`No evidence, no claim. No schema, no report. No review, no publish.`

即：一条分析结论（Claim）必须先有真实证据（Evidence）支撑，才允许进入报告；报告结构必须符合预定义的 Schema；每份报告必须经过质检 Agent 审核签字，才允许发布。

### 2.2 用户交互流程

```
用户输入需求（自然语言）
        ↓
  ① 研究计划生成（LLM 理解需求 → 竞品 + 维度 + 大纲）
        ↓
  ② 方案确认（用户可修改竞品列表、分析维度、报告章节）
        ↓
  ③ 开始分析（触发 17 步 DAG 工作流）
        ↓
  ④ 证据采集 → 抽取 → Claims 生成 → 质检（打回重做）
        ↓
  ⑤ 报告撰写（基于通过质检的 Claims）
        ↓
  ⑥ 查看报告（Markdown + HTML + 可视化图表）
```

## 三、工作流机制详解

### 3.1 17 步 DAG 全流程

```
① build_task_brief       构建任务简报（产品列表、目标）
② plan_schema            规划分析维度（功能/定价/用户画像等）
③ plan_sources           规划信息来源（官网/文档/社区等）
④ collect_sources        采集信息（爬虫或 replay 模式）
⑤ extract_evidence       从页面抽取结构化证据（实体/关系/数字）
⑥ evaluate_evidence      评估证据质量
⑦ pii_scrub              个人信息脱敏
⑧ extract_facts          抽取事实陈述
⑨ detect_schema_gaps     检测维度缺口（哪些维度缺证据）
⑩ coverage_critic        覆盖率评审（是否满足报告要求）
⑪ analyze_dimensions    分析师 Agent 生成 Claims（竞品对比断言）
⑫ review_claims          质检 Agent 审核 Claims
⑬ prepare_human_intervention 人工确认节点（软节点）
⑭ execute_rework         执行补证（coverage 不足时触发）
⑮ write_report_v2        报告撰写 Agent 生成章节（调用 deep_report.py）
⑯ final_review           终审（全局一致性检查）
⑰ export_report          导出报告（Markdown / HTML）
⑳ compute_metrics        计算质量指标（字数/覆盖率/review pass rate）
```

### 3.2 为什么用顺序执行而不是 LangGraph？

代码中通过 `force_sequential=True` 强制顺序执行，而不是用 LangGraph 的状态机。

原因：LangGraph 的 state 传播存在 bug——review_claims 修改了 signed_claims，但后续节点读取不到更新后的值。改用顺序 Python 执行，每步结果写入 WorkflowRepository，下一步直接读取，彻底绕过了 state 传播问题。

### 3.3 Agent 间结构化消息传递

#### 通信机制：WorkflowState 即消息总线

系统采用 **Hub-and-Spoke + TypedDict 消息总线** 架构，而非 LLM-level function calling。各 Agent 之间通过共享的 `WorkflowState`（TypedDict）传递结构化任务上下文，执行结果也以结构化 payload 形式写回状态字典，交给下一个 Agent。

```
Orchestrator
    │
    │ ① WorkflowState { run_id, products, task_brief, ... }
    ▼
CollectorAgent —→ { sources: [...], evidence_items: [...], errors: [] }
    │
    │ ② WorkflowState { sources, evidence_items, ... }
    ▼
Evaluator —→ { evidence_items: [...], evidence_evaluation: { usable_evidence, avg_score } }
    │
    │ ③ WorkflowState { evidence_items, evidence_evaluation, ... }
    ▼
AnalystAgent —→ { claim_drafts: [...], signed_claims: [] }
    │
    │ ④ WorkflowState { claim_drafts, ... }
    ▼
ReviewerAgent —→ { signed_claims: [...], rework_requests: [...] }
                                              │
                           质检打回 ← 如果有 claim 未达标
                                              │
    │ ⑤ WorkflowState { signed_claims, rework_requests, ... }
    ▼
ReworkAgent ← 从 rework_requests 读取打回指令，执行补证
    │
    │ ⑥ 回到 evaluate_evidence / analyze_dimensions，形成迭代闭环
    ▼
(循环直到所有 rework_requests 被解决，或达到 MAX_ITERATIONS 上限)
```

#### 质检打回：三层质量门控

| 门控层级 | 实现位置 | 门控逻辑 |
|---------|---------|---------|
| **证据质量门** | `EvidenceEvaluator` | 5 维度加权评分（相关性 25%、权威性 20%、时效性 10%、Schema 匹配 25%、信息密度 20%）；`usable_for_claim` 二值判定（final_score ≥ 0.32 **且** relevance ≥ 0.20 才允许用于声明）|
| **声明契约门** | `ReviewerAgent.review_claim()` | 8 项检查：来源类型白名单、PII 脱敏、置信度阈值、Schema 字段完整性、Evidence 可用性；不合格者 → `rework_required`，生成结构化 `rework_request` |
| **报告契约门** | `ReviewerAgent.review_report()` | 报告跨句与声明的链接完整性、引用有效性、PII 泄漏检测 |

打回消息结构示例（Reviewer → ReworkAgent）：

```python
state["rework_requests"].append({
    "dimension": "pricing_model",
    "claim_id": "claim_001",
    "reason": "UNUSABLE_EVIDENCE",     # 或 MISSING_EVIDENCE / PII_NOT_MASKED
    "evidence_ids": ["ev_001", "ev_002"],
    "priority": "high",
    "instructions": "Find pricing evidence from official_pricing_page for Coze",
})
```

#### 迭代闭环：三条反馈回路

系统有三条独立的迭代闭环，每条都有 `MAX_ITERATIONS` 硬上限防止无限循环：

**闭环 A — 覆盖率迭代**
```
coverage_critic → (覆盖率不足) → execute_rework → evaluate_evidence → analyze_dimensions
```
CoverageCritic 评估每个产品 × 维度的证据覆盖率，不足时生成补证任务，最多重试 3 轮。

**闭环 B — 声明迭代**
```
review_claims → (有 rework) → reflect_on_review → execute_rework → evaluate_evidence
```
Reviewer 质检未通过的 claim 生成 rework_request；reflect_on_review 用 LLM 进一步丰富打回指令的优先级和范围；ReworkAgent 补证后重新进入评估循环，最多 3 轮。

**闭环 C — 报告重写**
```
final_review → (rework_required) → write_report_v2 → final_review → export_report
```
终审发现报告质量问题（引用缺失、Schema 违规等）→ 触发报告重写，最多 3 轮。

所有闭环的数据都有 Before/After 对比记录在数据库中，可通过前端 DAG 视图实时观测。

```
coverage_critic 发现 Dify 维度覆盖率不足
        ↓
自动创建 rework 任务（rework_type=coverage_gap）
        ↓
用户在前端触发 execute_rework
        ↓
Collector 重新采集 → Evidence → Facts → Claims
        ↓
review_claims 重新审核
        ↓
coverage 更新，数据写入 DB（Before/After 对比）
```

## 四、Agent 设计

### 4.1 四大 Agent 职责

| Agent | 核心职责 | 关键工具 | 输入 | 输出 |
|-------|---------|---------|------|------|
| **Collector** | 采集 + 抽取 | HTTP 请求、正则、LLM 摘要 | URL 列表 | Evidence Items |
| **Analyst** | 竞品分析 | LLM 对比分析 | Evidence Items | Claim Drafts |
| **Reviewer** | 质检把关 | Evidence Contract 检查 | Claims + Evidence | Signed Claims / Rework |
| **Writer** | 报告撰写 | deep_report.py（章节生成） | Signed Claims | Markdown/HTML 章节 |

### 4.2 Reviewer 的 Evidence Contract 机制

每个分析维度有独立的来源白名单规则：

| 维度 | 只接受来源 | 拒绝来源 |
|------|----------|---------|
| pricing_model | `official_pricing_page` | blog、社交媒体 |
| function_tree | `documentation`、`github` | 新闻稿 |
| enterprise_readiness | `official_website`、`documentation` | 第三方文章 |

Reviewer 发现证据来源不符，直接将 Claim 降级为 `rework_required`，生成补证任务——这是真实的 feedback loop，不是模拟。

### 4.3 幻觉抑制三层防护

**第 1 层：引用强制**
- 每条 [E:N] 引用必须在 Evidence Appendix 中存在
- CitationVerifier 在报告导出前验证引用编号
- Gate-8 阻止引用不存在的 evidence

**第 2 层：内部一致性校验**
- Gate-1: 产品数量一致性
- Gate-2: 定价数字一致性
- Gate-4: Scorecard 非空
- Gate-7: POC 优先级合理性
- Gate-8: 引用编号有效性

**第 3 层：超长上下文分片**
- Evidence 超过 LLM context window 时自动分批
- 按 product 分组，保持相关 evidence 在同一批

## 五、研究计划动态生成机制

### 5.1 当前实现：LLM + Fallback 双轨

```
用户输入自然语言需求
        ↓
尝试调用 LLM（Doubao-Seed-2.0-lite，temperature=0.3）
        ↓
LLM 成功 → 合并 LLM 字段 + 模板结构 → generated_by="llm_augmented"
LLM 失败 → 降级到确定性模板 → generated_by="fallback"
```

**实际验证结果（2026-06-08 数据库记录）**：

刚才对 AskNews 和 Perplexity（两者均不在预定义列表中）调用研究计划生成：
- `generated_by = "llm_augmented"` ✅ LLM 调用成功
- `llm_call_succeeded = true`
- LLM 实际使用了：objective、competitor_selection_rationale、source_plan、schema_plan、workflow_plan
- 合并了 fallback 的结构：competitors、task_brief、source_discovery、report_outline、analysis_dimensions

### 5.2 竞品提取与 URL 自动发现机制

**预定义列表**（`KNOWN_COMPETITORS`，20+ 个产品）：
Dify、Coze、Flowise、LangGraph、LangChain、AutoGen、CrewAI、OpenAI Agents、Cursor、Windsurf、Trae、GitHub Copilot、Notion、Confluence、Coda、Slite、Airtable、ClickUp Docs、Slack、Microsoft Teams、Zoom、Google Meet

#### 未知产品不会完蛋——有多层兜底

实测 AskNews/Perplexity/AutoQL（均不在预定义列表）：

```
用户输入：对比分析 AskNews 和 Perplexity
        ↓
系统自动生成 discovery_queries（每个产品4条搜索查询）：
  AskNews → "AskNews official website" / "AskNews documentation pricing enterprise" / ...
  Perplexity → "Perplexity official website" / ...
        ↓
collect_sources 节点执行 URL 发现，兜底链：
  ① Doubao Web Search → 超时（火山引擎 API 不稳定）
  ② DuckDuckGo（免费，无需配置）→ ✅ 找到
        - AskNews: https://asknews.app/ ✅
        - Perplexity: https://www.perplexity.ai/ ✅
        - AutoQL: https://www.autoql.com ✅
  ③ LLM Inference（用模型知识推断URL）→ 再次兜底
  ④ Seed URLs → 已知产品用预定义URL
  ⑤ Fixture → 万不得已的静态数据
```

#### 自动发现链实测结果

| 产品 | DuckDuckGo 发现 | 发现的 URL |
|------|----------------|-----------|
| AskNews | ✅ | asknews.app, asknews.ai/docs |
| Perplexity | ✅ | perplexity.ai, perplexitycn.com |
| AutoQL | ✅ | autoql.com, npm (次要) |
| Dify | ✅ | dify.ai |
| Coze | ✅ | coze.cn, coze.cn/docs |

**结论**：用户懒得填网址不会完蛋。DuckDuckGo 搜索（免费，无需配置）是可靠的兜底。

#### 当前薄弱环节

| 薄弱点 | 现状 | 影响 |
|-------|------|------|
| Doubao Web Search 超时 | 超时后自动降级 DuckDuckGo | ✅ 兜底生效，不阻塞 |
| DuckDuckGo 在国内可能慢 | 国内访问 DuckDuckGo 有时不稳定 | ⚠️ 可能超时 |
| 采集后网页内容抓取 | 依赖 `fetch_url_with_fallback`（需要目标站可访问）| ⚠️ 目标站被墙则 evidence 质量差 |
| Replay 模式用 Fixture 数据 | Fixture 数据是预置的（dify.ai/coze.cn）| ⚠️ 新产品没有 Fixture |

#### 最坏情况分析

如果 DuckDuckGo 也超时，所有自动发现都失败：
- 竞品识别成功（name="AskNews"）✅
- 但 URL 发现失败 ❌
- `collect_sources` → `no seed_urls provided` → 创建人工干预节点
- 系统**不会崩溃**，而是**暂停等待用户补充 URL** ⚠️

这是设计上的主动阻断：与其生成一份没有 evidence 的空洞报告，不如等用户提供 URL。

### 5.3 Schema Type 推断：6 种模式动态切换

**实测 6 个不同查询的 schema 推断结果**：

| 查询 | 推断出的 schema_type | 维度数 | 章节数 |
|------|---------------------|-------|-------|
| Dify + Coze（Agent平台） | `ai_agent_platform` ✅ | 6 | 7 |
| Dify + AutoQL（数据平台） | `ai_agent_platform` ✅ | 6 | 7 |
| AskNews + Perplexity（AI搜索） | `competitor_landscape` ✅ | 6 | 3 |
| Notion + Confluence（知识管理） | `knowledge_management` ✅ | 9 | 12 |
| GitHub Copilot + Cursor（编程工具） | `ai_coding_assistant` ✅ | 6 | 7 |
| Slack + Teams（协作功能） | `pricing_analysis` ⚠️ | 6 | 13 |

**发现了一个 bug**：Slack + Teams 的查询意图是"协作功能对比"，但因为 Slack 被加入 pricing_analysis 的关键词列表，导致整个查询被误判为 `pricing_analysis`。

**Schema 推断优先级顺序**（`infer_schema_type` 代码中的检查顺序）：
```
1. knowledge_management（检查"notion/confluence/coda"...）
2. ai_coding_assistant（检查"cursor/copilot/windsurf"...）
3. ai_agent_platform（检查"dify/coze/rag/workflow"...）
4. pricing_analysis（检查"定价/pricing/slack/zoom"...）← Slack 触发这里
```

Slack/Zoom/Google Meet 在 pricing_analysis 的关键词里是为了检测"协作工具定价对比"，但当用户真正想做"协作功能对比"时会被误判。这需要在 `infer_schema_type` 中加入协作工具场景的判断，或调整关键词列表。

### 5.4 报告内容：完全动态生成

**实测 run_8e8343b559b94878_v2 报告内容**（34321 字）：

- 对比矩阵中的描述：**基于真实 evidence 生成**，不是模板占位符
  - `[E:1]` 引用编号对应 Evidence Appendix 中的真实 URL
  - "Supports visual definition of agentic workflows..." —— 这是 LLM 根据 evidence 原文生成的分析，不是"功能A vs 功能B"这种固定对比
- 每个 section ~600-1500 字，章节间内容不重复
- 包含具体产品名（Dify、Coze）、具体功能描述、具体证据引用
- 有 LLM 生成的解读（"> 解读: This comparison shows..."）

### 5.5 诚实评估：哪些是动态的，哪些是半静态的

| 环节 | 动态程度 | 证据 |
|------|---------|------|
| **LLM 调用研究计划** | ✅ 动态 | `generated_by=llm_augmented`，DB 记录为证 |
| **竞品识别** | ⚠️ 半动态 | 预定义 20+ 产品可自动补全 URL；未知产品提取但需用户补充 |
| **Schema Type 推断** | ✅ 动态 | AskNews→competitor_landscape vs Dify→ai_agent_platform，实证不同 |
| **分析维度选择** | ✅ 动态 | 6 种 schema_type 产生不同维度集（3-8 个维度） |
| **报告大纲** | ✅ 动态 | 6 种 outline 模板产生不同章节（3-13 个章节） |
| **报告正文内容** | ✅ 动态 | ~9000 字由 LLM 生成，含真实 evidence 引用，非占位符 |
| **Claims 生成** | ✅ 动态 | Analyst Agent 基于 Evidence 生成，12/12 全部签发 |
| **对比表内容** | ✅ 动态 | 表中 cell 内容基于 evidence 动态生成，不是"功能A：✓ 功能B：✗" |
| **SWOT 图表** | ✅ 动态 | chart_data 内容基于 Claims 动态生成 |
| **信息采集** | ⚠️ 半动态 | Replay 模式用预设数据；Real-time 模式真实爬虫 |

### 5.6 当前局限性

1. **Doubao Web Search 超时**：实测火山引擎的 DoubaoResponsesAPI 超时严重（30s timeout），自动降级 DuckDuckGo 兜底，但DuckDuckGo 国内访问也可能不稳定
2. **大纲模板选择**：从 6 个模板中选，不是 LLM 从零生成（在模板内已充分个性化）
3. **LLM 可能失败**：Rate Limit 时降级到模板，plan 仍可用但个性化程度降低
4. **Replay 模式 Fixture 数据局限**：Fixture 数据是预置的 dify.ai/coze.cn，Replay 模式不采集新产品的真实 evidence

## 六、亮点与创新

### 亮点 1：证据优先的 Claims 质检闭环

- 每条 Claim 必须经过 Reviewer Agent 签字
- 不合格的自动降级为 `rework_required`，生成结构化打回指令写入 `state["rework_requests"]`
- 质检结果通过 WorkflowState 消息总线传递给 ReworkAgent，形成完整的质检-打回-迭代闭环
- 数据库真实写入 Before/After 数字，可通过前端 DAG 视图实时观测迭代轮次

### 亮点 2：Schema 驱动的报告结构

- 6 种 Schema Type 对应 6 种报告结构
- 分析师只生成 schema 支持的 Claims
- 报告章节与研究维度一一对应

### 亮点 3：主动阻断机制

- Evidence Coverage < 80% 时，报告不输出误导性综合分
- 改为输出四级状态：Ready / Partial / Needs Verification / No Public Evidence

### 亮点 4：可观测工作流

- 17 个节点每步状态写入 WorkflowRepository
- 支持从任意节点断点重跑
- LLM 调用全程 Trace 记录
