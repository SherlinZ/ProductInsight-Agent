## 根因总结

### 为什么之前报告只有 500 字？

| 原因 | 解决方案 |
|------|---------|
| ALLOWED_DIMENSIONS 白名单不包含 analyst 实际生成的维度 | 扩展白名单（已修） |
| fallback LLM 调用用了未定义变量 api_key | 改用 get_llm_client()（已修） |
| product_id 被错误拼接 | 从 task_brief.products 取正确名称（已修） |
| collect_sources 每次都走 real_time 路径 | replay 时强制 mode=replay（已修） |

### 为什么现在报告有内容了？

- signed_claims: 0 → 12 条（全部签发，rework_required=0）
- 总字数：~500 → ~9000 字
- 有功能对比矩阵、SWOT、用户场景、定价对比、TCO 模型等结构化内容

## 剩余问题

### P1. 对比表数量略低于目标（3 vs 4）

当前生成了3张表（功能对比、定价对比、用户场景），距离目标4张差1张。
影响不大，但可以补充一张"市场定位对比"或"集成能力对比"表。

### P2. 部分维度无 Claims（multi_agent, user_persona 等）

Dify 的 multi_agent、pricing_model 和 Coze 的 user_persona、multi_agent 维度没有生成 Claims，
可能是 analyst agent 的 prompt 没有覆盖这些维度。
影响较小（覆盖率72.9%已达标），可在 analyst prompt 中补充。

### P3. API Rate Limit（TPM exceeded）— 历史问题

2026-05-29 时遇到过火山引擎 TPM 超限，但 2026-06-07 跑 v2 报告时未复现。
**当前状态：正常**。

### P4. Section review 太严格导致 rework 循环 — 历史问题

v2 报告 12/12 claims 全部签发，rework_required=0，说明 reviewer 已经可以正常签发，
不再触发重复 rework 循环。**当前状态：已解决**。
