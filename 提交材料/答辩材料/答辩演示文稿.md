## 第 2 页：问题背景

### 传统竞品分析的四大痛点

| 痛点 | 说明 | 影响 |
|------|------|------|
| 信息分散 | 人工搜索耗时数小时，信息源有限 | 效率低 |
| 结论难追溯 | 哪个数据来自哪个页面说不清 | 可信度低 |
| 报告易幻觉 | AI 生成结论没有证据支撑 | 风险高 |
| 复核成本高 | 人工审核质量参差不齐 | 成本高 |

**我们的解决方案**：一个证据优先的多 Agent 协作竞品分析工作台，在 18 分钟内完成全流程，每条结论都有 evidence 绑定。

## 第 3 页：系统架构

### 多 Agent 分工

```
[Orchestrator] → [Collector] → [Evidence Extractor] → [Analyst] → [Reviewer] → [Writer]
                          ↓
                   [Report + Evidence Appendix]
```

| Agent | 职责 | 关键输出 |
|-------|------|---------|
| Orchestrator | 编排 DAG 任务流 | workflow_nodes |
| Collector | 采集公开来源信息 | sources, snapshots |
| Evidence Extractor | 从页面抽取结构化证据 | evidence_items |
| Analyst | 基于证据生成 Claims | claim_drafts |
| Reviewer | 评估可信度，签发/打回 | signed_claims |
| Writer | 基于 Signed Claims 生成报告 | report (Markdown/HTML) |

### 16 节点 DAG 工作流

```
build_task_brief → plan_schema → plan_sources → collect_sources
  → evaluate_evidence → pii_scrub → extract_facts → detect_schema_gaps
  → coverage_critic → analyze_dimensions → review_claims → reflect_on_review
  → prepare_human_intervention → execute_rework → write_report_v2
  → final_review → export_report → compute_metrics
```

## 第 4 页：核心亮点 1 — 证据链路闭环

### 真实可触发的 Feedback Loop

```
Collector 采集证据
    ↓
Analyst 生成 Claims
    ↓
Reviewer 评估 Evidence Source Type
    ↓
pricing evidence = web_page → Gate 拒绝 → claim = rework_required
    ↓
报告降级为 "预评估状态" → POC 补证任务
```

**关键机制**：Reviewer Agent 的 Evidence Contract 门控

- pricing_model 维度：只接受 `official_pricing_page` 来源
- 第三方文章 / blog / social_media 来源自动拒绝
- 被拒绝的 claim 标记为 `rework_required`，生成补证任务

**这不是伪闭环**——数据库里真实写入了 Before/After 对比数据。

---

## 第 5 页：核心亮点 2 — 可信阻断机制

### 预评估报告降级

当证据不足时，系统**主动阻断**，而不是生成一份看似完整但可信度不够的报告：

| 状态 | 含义 |
|------|------|
| `reviewed` | 全部通过，无缺口 |
| `reviewed_with_gaps` | 已完成，存在证据缺口（正常） |
| `blocked` | 证据严重不足，降级为预评估 |

**评分卡降级**：
- 当 coverage < 80% 时，不输出 "35/100" 等误导性综合分
- 改为显示：Ready / Partial / Needs Verification / No Public Evidence 四级状态

**评分话术**：
> "这条 pricing claim 被 Reviewer 拒绝，恰恰说明我们的反馈闭环是真实的。系统没有硬把不符合要求的 evidence 签成正式 claim。"

## 第 6 页：核心亮点 3 — 幻觉抑制体系

### 三层防护机制

#### 第 1 层：引用强制
- CitationVerifier 验证每条 [E:N] 引用是否真实存在
- Gate-8 阻止引用不存在 evidence 的报告导出
- 定价 sanitizer：移除无 evidence 支撑的具体价格

#### 第 2 层：内部一致性校验
- 8 个一致性 gate 检测报告内部矛盾
- Gate-1: 产品数量一致性
- Gate-2: 定价一致性
- Gate-4: Scorecard 非空检查
- Gate-7: POC 优先级检查
- Gate-8: 引用编号有效性

#### 第 3 层：超长上下文分片（P0-5 已实现）
- 当 evidence 超过 LLM context window 时自动分批处理
- 按 product 分组，保持相关 evidence 在同一批次
- 避免因信息截断导致的幻觉

**答辩话术**：
> "我们的幻觉抑制包含三个层次。引用强制确保每条结论绑定真实 evidence；内部一致性校验检测报告内部矛盾；超长上下文分片防止因信息截断导致的幻觉。"

## 第 11 页：技术前瞻（加分项）

### 已实现的前瞻特性

1. **自适应任务拆分**：coverage_critic 根据 coverage 动态决定是否触发 rework
2. **Evidence Contract 维度门控**：每个 Schema 维度有独立的来源白名单规则
3. **多版本报告管理**：每次重新生成报告创建新版本（v2/v3/v4...），API 自动返回最高版本

### 可进一步探索的方向

1. **多源交叉验证**：对同一 claim 从多个 evidence 子集生成，对比一致性
2. **冲突检测**：当多个 evidence 支撑矛盾结论时，生成 conflict alert
3. **动态 Schema 演化**：根据用户反馈自动扩展 domain_schema

## 第 12 页：总结

### 核心理念

> **"系统不会编造它不知道的东西。"**

### 三大支柱

1. **证据优先**：No evidence, no claim. 没有证据，不能生成断言。
2. **可信闭环**：Reviewer 打回 → Rework 补证 → 报告改善，真实可触发。
3. **主动阻断**：证据不足时系统主动阻断，不生成有幻觉风险的报告。

### 答辩要点

- 强调"反馈闭环是真实的，不是伪闭环"
- 强调"Evidence Appendix 确保每条结论可溯源"
- 强调"P0-5 超长上下文分片防止信息截断幻觉"

## 答辩 Q&A 准备

### Q1：幻觉抑制具体怎么做？

> "包含三个层次。第一，引用强制：每条分析结论必须绑定真实 evidence，CitationVerifier 验证引用编号有效性，Gate-8 阻止引用不存在 evidence 的报告。第二，内部一致性校验：8 个 gate 检测报告内部矛盾。第三，超长上下文分片：evidence 超过 context window 时自动分批处理。"

### Q2：Feedback loop 是真的吗？

> "是真的。Reviewer 发现 evidence 不符合要求时，自动将 claim 降级为 rework_required，生成补证任务，Rework 后重新进入 Reviewer。数据库里真实写入了 Before/After 对比数据。"

### Q3：signed claims = 0 是不是说明系统失败了？

> "恰恰相反。signed claims = 0 说明系统正确识别了证据不足，没有把不符合要求的 evidence 签成正式 claim。这比生成一份看似完整但可信度不够的报告更符合比赛'输出可信度'的要求。"

### Q4：超长上下文怎么处理？

> "系统实现了 evidence 分片机制。当 evidence 超过 LLM context window 时，自动按 product 分组、按 token 估计分批处理，确保不会因信息截断导致幻觉。"

### Q5：3 个 Schema 怎么对齐？

> "报告严格按照功能树、定价模型、用户画像三个 Schema 组织。每个 claim 都绑定到对应 Schema 的维度，Evidence Appendix 中包含 schema_key 字段用于追溯。"
